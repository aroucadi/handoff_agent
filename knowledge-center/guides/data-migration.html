Data Migration Guide

Migrate from Clarity CA PPM, Microsoft Project Server, Jira Portfolio, or spreadsheet-based PPM to ClawdView with minimal disruption and zero data loss.

Migration Framework

ClawdView migrations follow a proven 5-phase framework that has been validated across 200+ enterprise deployments:

Phase Duration Key Activities
1. Discovery 1-2 weeks Source system audit, data quality assessment, scope definition
2. Mapping 1-2 weeks Field mapping, data transformation rules, custom field creation in ClawdView
3. Trial Migration 1 week Migrate 10-20% of data to validate mappings and transformations
4. Full Migration 1-3 days Complete data extraction, transformation, and load with parallel running
5. Validation 1-2 weeks Data integrity checks, user acceptance testing, rollback readiness

Source System: Clarity CA PPM

ClawdView Migration Accelerator

The Migration Accelerator is a pre-built ETL tool specifically designed for Clarity → ClawdView migrations. It includes:

  • Pre-configured field mappings for Clarity's standard data model (projects, resources, timesheets, financials)
  • OBS (Organizational Breakdown Structure) → ClawdView workspace hierarchy converter
  • Process step → Stage-gate mapping templates
  • Financial period actuals migration with currency conversion support
  • Custom attribute and lookup table migration

Typical migration timelines from Clarity:

  • Projects + Resources: 2-4 weeks (includes testing)
  • Financial History: Additional 2-3 weeks (high-fidelity, auditable)
  • Timesheets: Additional 1-2 weeks (optional — most clients start fresh)
⚠ Common Pitfall

Clarity custom attributes often have inconsistent data types across instances (e.g., a "Priority" field stored as text in one project and as a lookup in another). The Migration Accelerator detects these inconsistencies during Discovery phase and generates a remediation report. Address data quality issues before the trial migration, not during.

Source System: Microsoft Project Server / Project Online

Migrations from Microsoft Project environments involve these key considerations:

  • MPP Files: PPM Pro natively imports .mpp files with full WBS, dependencies, and resource assignments. Batch import available via the API.
  • Server Fields: Custom enterprise fields map to ClawdView custom fields. The field type mapping handles text, lookup, date, number, and formula fields.
  • Resource Pool: The centralized resource pool migrates to PPM Pro's Resource Management module. Cost rates, availability calendars, and skill profiles are preserved.
  • Baselines: Up to 11 baselines per project are migrated. ClawdView preserves baseline snapshots for variance analysis post-migration.

Source System: Jira Portfolio / Jira Align

For organizations using Jira at scale, two migration strategies exist:

Strategy A: Coexistence (Recommended)

Keep Jira for team-level execution and use ClawdView for portfolio/program management. Use Hub connectors for bidirectional sync. This avoids disrupting developer workflows.

Strategy B: Full Migration

Migrate portfolio-level data (Initiatives, Epics, Themes) to ClawdView Portfolios. Map Jira Align Program Increments to AgilePlace PI structure. Team-level stories remain in Jira with Hub sync.

Source System: Spreadsheets

ClawdView provides a Spreadsheet Import Wizard that handles:

  • Excel (.xlsx) and CSV file ingestion
  • Column-to-field auto-mapping with manual override
  • Date format detection and normalization
  • Hierarchical relationship inference (via indentation or ID columns)
  • Duplicate detection across imported batches
💡 Best Practice

Before migrating from spreadsheets, spend one week standardizing your data model. Create a master template with consistent column names, date formats, and status values. Migrations from well-structured spreadsheets complete in hours; migrations from inconsistent spreadsheets take weeks of cleanup.

Rollback Strategy

Every ClawdView migration includes a documented rollback path:

  • Pre-migration snapshot: Full data export from ClawdView before migration (preserves any manually-created content)
  • Source system preservation: Source system remains operational for 90 days post-migration (read-only mode if licensing allows)
  • Reverse ETL: ClawdView data can be exported back to source format if rollback is needed within 30 days
