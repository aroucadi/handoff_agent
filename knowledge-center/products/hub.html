ClawdView Hub

The integration and workflow engine connecting ClawdView with your existing enterprise tools — Jira, ServiceNow, SAP, Azure DevOps, Salesforce, and 100+ pre-built connectors. No code required.

Overview

ClawdView Hub is the integration backbone of the ClawdView platform. It enables bidirectional data flow between ClawdView modules (Portfolios, AgilePlace, PPM Pro) and external enterprise systems. Hub uses a connector-based architecture where each integration is a configured connector with field mapping, sync rules, and conflict resolution policies.

Hub eliminates the need for custom middleware or point-to-point integrations. Organizations typically reduce integration maintenance effort by 60-80% compared to custom API scripts by centralizing all data flows through Hub's managed connectors.

Architecture

Hub operates on a three-tier architecture:

Connector Layer

Each connector encapsulates the logic for communicating with a specific external system. Connectors handle authentication (OAuth 2.0, API keys, SAML tokens), pagination, rate limiting, and error retry. The connector SDK allows customer-developed connectors for proprietary systems.

Mapping Engine

The visual field mapper allows administrators to define how data transforms between ClawdView and external systems. Features include:

  • Direct Mapping: 1:1 field mapping (e.g., Jira "Story Points" → AgilePlace "Effort")
  • Value Transformation: Lookup tables, formulas, and conditional logic (e.g., Jira priority "P1" → ClawdView priority "Critical")
  • Calculated Fields: Derive new values from multiple source fields
  • Default Values: Provide fallbacks when source data is missing
  • Custom Scripts: JavaScript-based transformation functions for complex logic

Sync Engine

The sync engine manages data flow with these capabilities:

  • Near Real-Time: Webhook-triggered sync within 60 seconds for supported connectors
  • Scheduled Batch: Configurable intervals (5 min to 24 hours) for polling-based connectors
  • Conflict Resolution: Last-writer-wins, source-system-wins, or manual review policies
  • Audit Trail: Full sync history with field-level change tracking
  • Retry & Dead Letter: Automatic retry with exponential backoff. Failed records routed to review queue
⚠ Rate Limits

Hub respects external system rate limits. For Jira Cloud, this means a maximum of 100 requests per 60 seconds per connector. Large initial syncs (>10,000 items) are automatically throttled and may take 2-4 hours. Plan initial sync during off-peak hours. For Jira Data Center, limits are configurable on the Jira side.

Common Integration Patterns

Jira ↔ AgilePlace Bidirectional Sync

The most common Hub pattern. Development teams continue using Jira for sprint execution, while program managers use AgilePlace for cross-team coordination. Hub syncs epics and stories bidirectionally, keeping both systems as sources of truth for their respective audiences. Recommended field mapping:

  • Jira Epic → AgilePlace Feature card (with children aggregation)
  • Jira Story → AgilePlace Story card (status, assignee, estimate, labels)
  • Sprint boundaries → AgilePlace iteration lanes
  • Avoid: Sub-task sync, comment sync (creates noise and API load)

SAP Financial Actuals → PPM Pro

Nightly batch sync of actual costs from SAP cost centers to PPM Pro projects. Mapping uses WBS element IDs as the join key. Hub transforms SAP's cost element hierarchy into PPM Pro's budget categories. Typical data volume: 10,000-50,000 cost line items per sync cycle.

ServiceNow Incident → AgilePlace Card

Production incidents in ServiceNow auto-create cards on a "Production Support" board in AgilePlace. When the AgilePlace card is moved to "Done," Hub updates the ServiceNow incident to "Resolved." This creates a unified view of planned work and unplanned production support within the same team board.

💡 Integration Architecture Tip

Always designate a system of record for each data entity. For example: Jira is system of record for developer-facing story details, while AgilePlace is system of record for program-level metadata (PI assignments, dependencies, business value). This principle prevents the most common integration failure: update loops where both systems write the same field and trigger infinite sync cycles.

Connector Development SDK

For proprietary systems not in the pre-built catalog, Hub provides a Connector SDK (TypeScript/Node.js):

// Example: Custom connector for internal asset management system export class AssetConnector extends BaseConnector { async fetchItems(since: Date): Promise<Item[]> { ... } async pushItem(item: Item): Promise<void> { ... } getFieldSchema(): FieldSchema[] { ... } }

Custom connectors are deployed as serverless functions on Google Cloud Functions and registered in Hub's admin console. The SDK handles authentication, retry logic, and error reporting — developers only implement data fetch/push logic.

Prerequisites

  • ClawdView Platform license with Hub module
  • Admin access to target external systems for API credentials
  • Network connectivity: Hub operates from GCP (us-central1, eu-west1, ap-southeast1); ensure firewall rules allow outbound HTTPS
  • For custom connectors: Node.js 18+ development environment and GCP project access
